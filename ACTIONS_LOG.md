## 2026-06-16: Base44 + Supabase hardening

Date: 2026-06-16

## Context
The live GradPilot AI product runs on **Base44** (app id `69c7f5789f546b6e6427a219`),
with a **Supabase** Postgres mirror (`tqpsvhtnoqhrvkntzlvz`) and Stripe billing.
This git repo previously had no committed code; it now holds the audit report and
the database hardening migrations as a backup.

## Done

### 1. Site audit (committed)
- `GradPilot_Site_Audit_2026-06-16.pdf` — full audit, progress report, weekly plan.

### 2. Supabase security + performance hardening (APPLIED — migration 0001)
Applied `supabase/migrations/0001_security_perf_hardening.sql` to the live DB.
- Pinned `search_path` on `set_updated_at` (fixed mutable search_path warning).
- Revoked RPC EXECUTE on `handle_new_user()` (trigger-only) from public/anon/authenticated.
- Revoked `is_admin()` EXECUTE from anon/public; granted to authenticated only.
- Replaced the always-true waitlist INSERT policy with a validated insert
  (non-empty name + email format check).
- Split every `FOR ALL` policy into per-command policies, removing duplicate
  permissive policies.
- Wrapped `auth.uid()` / `is_admin()` in scalar subqueries across all policies
  (fixes auth_rls_initplan re-evaluation).
- Added 8 covering indexes for previously unindexed foreign keys.

**Result:** security advisors 6 -> 1; performance lints 169 -> 16.

### 3. Blog posts published (Base44)
Set `isPublished=true` and `publishedAt=2026-06-16` on the 5 draft posts
(graduate schemes, Skilled Worker vs Graduate Route, UK bank account,
salary expectations, NHS registration). Blog now has 10 live posts.

## Remaining advisor items (intentional / low value)
- `is_admin` executable by **authenticated** — REQUIRED: RLS policies call it, so
  signed-in users must be able to execute it. By design.
- 2 `multiple_permissive_policies` (testimonials, room_listings, authenticated SELECT)
  — deliberate trade-off to keep `is_admin` un-executable by anon.
- 11 `unused_index` (INFO) — expected on empty tables; indexes will be used once
  data and queries exist. Do not drop.
- 2 `auth_rls_initplan` on `usage_tracking` (`usage_insert`, `usage_update`) — these
  policies are generated by Base44's own RLS sync. An optional fix is provided in
  `supabase/migrations/0002_usage_tracking_initplan.sql` (NOT applied; Base44 may
  re-sync and revert it).

## Could not do with current tools
- Delete the duplicate empty Base44 app (`6a20686693a7ad3e4c87876a`) — no delete tool.
- Hard-delete sample rows (example.com waitlist, sample mentors) — no delete tool.
- These require action in the Base44 dashboard.

## Key finding
Base44 actively manages/re-syncs RLS on its tables (new `usage_tracking` write
policies reappeared after migration 0001). Treat **Base44 as the source of truth**;
direct Supabase schema changes to Base44-managed tables may not fully persist.
